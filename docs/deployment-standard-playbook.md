# 标准化产品部署流程

## 目的
把“新产品部署到云服务器”这件事沉淀成一套可重复执行的标准流程，减少每次部署时反复问同样的问题，也避免在未确认线上环境前直接覆盖现有服务。

本流程适用于：
- 腾讯云 `CVM`
- 阿里云 `ECS`
- 华为云 `ECS`
- 其他 Linux 单机云服务器

## 标准流程总览
1. 明确部署目标
2. 收集接入信息
3. 远端环境预检
4. 冲突与部署位点决策
5. 项目部署适配
6. 上线执行
7. 验收与回滚
8. 生产化补齐

---

## 1. 明确部署目标
先确认这次部署属于哪一种：

### A. 演示版
特点：
- 优先快速可访问
- 可先使用本地文件或轻量存储
- 可以接受单机、低并发

### B. 内测版
特点：
- 需要相对稳定
- 开始考虑数据持久化和备份
- 需要日志和基础监控

### C. 正式生产版
特点：
- 需要数据库
- 需要对象存储
- 需要鉴权/权限
- 需要监控、备份、回滚
- 需要严格安全策略

如果没有明确说明，默认先按 `演示版` 上线。

---

## 2. 收集接入信息
每次部署前必须先拿到这些信息。

### 基础信息
- 公网 IP
- 云厂商
- 服务器类型
- 操作系统版本
- SSH 用户名
- SSH 登录方式
- SSH 端口

### 访问入口信息
- 主域名
- 是否已解析
- 是否已有子域名
- 是否已有 SSL 证书
- 是否允许替换现有站点

### 业务运行信息
- 是否接真实模型 API
- 是否已有数据库
- 是否已有对象存储
- 是否已有 Redis
- 预期用户量
- 是否已有现网业务在同一台机器上运行

### 最小信息模板
见 `docs/deployment-intake-template.md`

---

## 3. 远端环境预检
在真正部署前，必须先检查远端机器状态。

### 必查项
- 操作系统
- 磁盘空间
- 内存
- Docker / Docker Compose
- 80/443/22 端口占用
- 现有容器
- 现有 Nginx / Caddy / Apache
- 现有站点目录
- 安全组是否放行

### 标准原则
- 没做预检，不允许直接部署
- 没确认端口占用，不允许直接绑定 80/443
- 没确认是否已有线上业务，不允许替换入口

### 预检脚本
见：
- `deploy/standard/server-preflight.sh`

---

## 4. 冲突与部署位点决策
预检后要先决定“部署到哪里”，而不是直接启动容器。

### 场景 1：机器干净
可以直接：
- 80/443
- 主域名

### 场景 2：机器已有现有服务
必须先做决策：

1. 替换现有站点
2. 使用新子域名
3. 使用临时端口
4. 使用独立新机器

### 推荐顺序
1. 新子域名
2. 临时端口
3. 替换现有站点

因为“替换现有站点”风险最高。

---

## 5. 项目部署适配
确定部署位点后，再让项目变成可部署形态。

### 最小要求
- 前端支持生产 API 地址配置
- 后端支持环境变量
- 数据目录可配置
- CORS 可配置
- Dockerfile
- docker-compose
- `.env.example`
- 部署文档

### 演示版可接受
- JSON 文件持久化
- 单机线程任务
- 同机前后端

### 正式版必须补齐
- 数据库
- 文件存储
- 鉴权
- 队列
- 监控
- 备份

---

## 6. 上线执行
上线执行建议固定为以下顺序。

1. 上传代码或拉取代码
2. 复制 `.env.example` 为 `.env`
3. 填写环境变量
4. 构建镜像
5. 启动容器
6. 看容器状态
7. 看应用日志
8. 健康检查
9. 浏览器访问验证

### 严禁
- 未检查日志就宣布部署完成
- 未做健康检查就切流量
- 未确认端口冲突就启动

---

## 7. 验收与回滚
每次部署都必须有可验证结果和回滚方案。

### 最小验收项
- 首页可访问
- 健康检查返回正常
- 前后端通信正常
- 上传和主链路功能可跑通
- 容器处于 `Up`

### 最小回滚方案
- 保留旧容器
- 保留旧镜像 tag
- 保留旧配置
- 失败时停止新容器并恢复原入口

---

## 8. 生产化补齐
演示版上线后，如果要进入正式使用阶段，建议按这个顺序演进：

1. 数据库替换本地 JSON
2. COS 替换本地文件
3. 用户与权限
4. 队列与异步任务
5. 监控告警
6. HTTPS 与证书自动续期
7. 备份与恢复演练

---

## 标准产物清单
以后每个新项目，部署前都应该至少产出这些文件：

- `Dockerfile`
- `docker-compose.yml`
- `.env.example`
- 部署文档
- 服务器预检脚本
- 上线验收清单

---

## 本次经验提炼
这次 JudgeWrite 部署里最关键的经验有两条：

1. 用户最开始给出的 SSH 用户名是错的，必须先验证登录再继续。
2. 服务器上已经有现有业务在跑，所以在没有确认是否替换现有站点前，不能直接占用 `80/443`。

这两条以后都应视为标准检查项，而不是临时发现的问题。
